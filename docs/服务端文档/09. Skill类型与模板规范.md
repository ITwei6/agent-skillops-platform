# 09. Skill类型与模板规范

## 一、文档目的

不同类型的 Skill 需要不同结构。

如果所有 Skill 都只是一段自由文本，后续很难审核、搜索、维护和复用。

因此平台需要定义 Skill 类型，并为不同类型提供模板规范。

## 二、Skill 类型

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| onboarding | 项目入门 | 新成员理解项目结构 |
| troubleshooting | 排错处理 | Docker SSH 连接失败排查 |
| workflow | 开发流程 | 新增一个子服务的标准流程 |
| review | 代码审查 | C++ 微服务代码审查清单 |
| deployment | 部署运维 | 服务构建和部署步骤 |
| testing | 测试验证 | 接口回归测试流程 |
| documentation | 文档生成 | 生成接口文档或设计文档 |
| convention | 规范约束 | 项目目录、命名、提交规范 |
| integration | 第三方集成 | 接入 RabbitMQ/Redis/FastDFS |

## 三、通用 Skill 结构

所有 Skill 至少包含：

```text
---
name: skill-name
description: Use when ...
---

# Skill Name

## When To Use

## Inputs

## Workflow

## Output

## Quality Checks

## Boundaries
```

## 四、类型模板

### 4.1 onboarding 项目入门类

适用场景：

- 新成员加入项目
- 第一次分析陌生仓库
- 需要快速理解架构

必须包含：

```text
项目背景
目录结构
核心服务
启动流程
关键配置
常见入口文件
阅读顺序
```

审核重点：

- 是否和当前项目版本一致
- 是否覆盖真实入口
- 是否避免泛泛而谈

### 4.2 troubleshooting 排错类

适用场景：

- 某类问题重复出现
- 有明确现象和排查路径
- 有可验证解决方案

必须包含：

```text
问题现象
可能原因
排查步骤
修复方案
验证方式
风险提醒
```

审核重点：

- 是否有实际验证记录
- 是否包含危险命令
- 是否区分现象和猜测

### 4.3 workflow 开发流程类

适用场景：

- 新增模块
- 新增接口
- 新增子服务
- 固定开发流程

必须包含：

```text
前置条件
步骤顺序
涉及文件
检查点
完成标准
不应修改的内容
```

审核重点：

- 是否符合项目工程规范
- 是否避免越权修改
- 是否有明确完成标准

### 4.4 review 代码审查类

适用场景：

- 指导 Codex 或团队成员做代码 review
- 检查特定技术栈的常见问题

必须包含：

```text
审查范围
重点检查项
风险等级
输出格式
不处理范围
```

审核重点：

- 是否聚焦真实风险
- 是否避免风格偏好当作缺陷
- 是否要求引用文件和行号

### 4.5 deployment 部署运维类

适用场景：

- 环境搭建
- 服务部署
- 容器启动
- 运行检查

必须包含：

```text
目标环境
依赖服务
配置项
启动步骤
健康检查
回滚方案
风险命令说明
```

审核重点：

- 是否包含生产风险
- 是否明确环境边界
- 是否有回滚方式

### 4.6 testing 测试验证类

适用场景：

- 接口测试
- 回归测试
- 子服务验收

必须包含：

```text
测试目标
测试数据
测试步骤
预期结果
失败排查
覆盖范围
```

审核重点：

- 是否可复现
- 是否有明确断言
- 是否覆盖关键路径

### 4.7 convention 规范约束类

适用场景：

- 代码规范
- 目录规范
- 提交流程
- 文档规范

必须包含：

```text
适用范围
强制规则
推荐规则
反例
例外情况
检查方式
```

审核重点：

- 是否和团队规范一致
- 是否有清晰边界
- 是否避免过度约束

## 五、模板选择规则

平台在生成 Skill 草稿时，应根据经验内容自动推荐类型。

规则示例：

- 包含“报错、失败、无法连接、排查” -> troubleshooting
- 包含“新增、开发流程、实现步骤” -> workflow
- 包含“部署、启动、Docker、环境” -> deployment
- 包含“review、检查、风险、代码质量” -> review
- 包含“新成员、项目结构、入门” -> onboarding

生成后允许提交人手动修改类型。

## 六、类型与审核关系

不同类型默认风险不同：

| 类型 | 默认风险 |
| --- | --- |
| onboarding | low |
| documentation | low |
| convention | medium |
| workflow | medium |
| testing | medium |
| troubleshooting | medium |
| deployment | high |
| integration | high |
| review | medium |

最终风险等级以自动检查和人工审核结果为准。
