# 08. Skill质量标准与审核规范

## 一、文档目的

本平台的核心不是“生成 Skill”，而是让团队沉淀出可靠、可复用、可维护的 Skill。

因此必须建立统一的 Skill 质量标准，避免审核完全依赖个人经验，避免团队 Skill 仓库变成低质量知识堆积。

## 二、质量评价维度

### 2.1 场景明确性

检查 Skill 是否说明清楚：

- 什么时候使用
- 面向什么任务
- 适用于哪个项目或技术栈
- 不适用于哪些场景

不合格示例：

```text
Use this skill when doing backend development.
```

合格示例：

```text
Use this skill when diagnosing SSH connection failures from Windows VS Code to the Docker-based C++ microservice development environment.
```

### 2.2 触发描述准确性

Skill 的 description 决定 Codex 什么时候使用它。

要求：

- 不得过宽
- 不得过窄到无法匹配真实任务
- 不得包含一次性上下文
- 不得把多个无关场景混在一起

### 2.3 内容可复现性

Skill 应该能指导另一个成员复现工作流程。

必须包含：

- 前置条件
- 操作步骤
- 判断标准
- 输出结果
- 异常处理

### 2.4 来源可信度

Skill 不能只来自一次模型猜测。

可信来源包括：

- 已验证的排错过程
- 项目文档
- 代码仓库中的真实配置
- PR/Issue 讨论
- 技术负责人确认
- 多次重复出现的问题

### 2.5 安全性

必须检查：

- 是否含有密钥
- 是否含有账号密码
- 是否含有 token
- 是否含有未脱敏内网地址
- 是否含有危险命令
- 是否包含未授权客户信息

### 2.6 维护性

Skill 应该便于后续更新。

要求：

- 结构清晰
- references 和 scripts 分离
- 不把大量长文堆在 `SKILL.md`
- 明确适用版本
- 有变更记录

## 三、评分模型

每个 Skill 草稿自动生成一份质量评分。

| 维度 | 分值 | 说明 |
| --- | --- | --- |
| 场景明确性 | 20 | 是否清楚说明适用场景 |
| 触发描述准确性 | 20 | description 是否合适 |
| 可复现性 | 20 | 是否能按步骤执行 |
| 来源可信度 | 15 | 是否有可靠来源 |
| 安全性 | 15 | 是否无敏感和危险内容 |
| 维护性 | 10 | 是否易维护和版本化 |

总分：

```text
90-100：优秀，可进入快速审核
75-89：合格，可进入普通审核
60-74：需修改
0-59：不建议发布
```

## 四、风险等级

### 4.1 low

特点：

- 无敏感信息
- 无危险命令
- 场景明确
- 有参考来源

处理：

- 普通 reviewer 审核即可

### 4.2 medium

特点：

- 存在触发范围不够精确
- references 不充分
- 与已有 Skill 有一定相似
- 包含需要人工确认的操作步骤

处理：

- 至少一名 reviewer 审核通过
- 必须处理或确认风险项

### 4.3 high

特点：

- 涉及部署、权限、删除、重置、生产环境
- 包含脚本
- 包含高风险命令
- 涉及公司内部系统或敏感路径

处理：

- reviewer + maintainer 双人确认
- 必须保留审核理由

### 4.4 blocked

特点：

- 明文密钥
- 明文密码
- 未脱敏 token
- 未授权客户数据
- 无法解释的危险脚本

处理：

- 禁止发布
- 必须退回修改

## 五、审核检查清单

Reviewer 审核时必须检查：

```text
1. Skill 名称是否清晰
2. description 是否准确
3. 是否说明适用场景
4. 是否说明不适用场景
5. 是否包含敏感信息
6. 是否包含危险命令
7. 是否有 references 支撑
8. 是否和已有 Skill 重复
9. 是否符合类型模板
10. 是否需要拆分成多个 Skill
11. 是否需要补充验证步骤
12. 是否需要标记项目或版本范围
```

## 六、审核决策标准

### 6.1 通过

满足：

- 无 blocked 风险
- 质量评分达到发布要求
- 审核人确认内容正确
- 状态流转符合要求

### 6.2 要求修改

适用：

- 触发描述不清楚
- 缺少 references
- 步骤不可复现
- 有重复风险但可合并
- 安全风险可以通过脱敏解决

### 6.3 驳回

适用：

- 内容价值不足
- 不具备复用性
- 只是一次性聊天记录
- 缺乏验证依据
- 与已有 Skill 完全重复

### 6.4 阻塞

适用：

- 敏感信息未处理
- 危险脚本无法解释
- 违反团队安全策略

## 七、平台自动检查项

自动检查不替代人工审核，只提供辅助判断。

自动检查包括：

- 敏感信息扫描
- 危险命令扫描
- description 范围检查
- Skill 格式检查
- 相似 Skill 检索
- references 缺失检查
- 类型模板匹配检查
