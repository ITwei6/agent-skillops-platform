# 06. 开发规范与工程约束

## 一、文档目的

本文用于约束后续代码实现，防止项目在实现阶段出现目录混乱、服务职责不清、接口随意变更、数据模型失控等问题。

本项目后续代码由实现窗口负责，本窗口只负责项目设计、架构评审和规范制定。

## 二、总体工程原则

### 2.1 先设计后实现

任何子服务开始编码前，必须先完成：

1. 子服务职责说明
2. 对外接口说明
3. 内部 RPC 说明
4. 数据表或缓存依赖说明
5. MQ 消息依赖说明
6. 错误码说明
7. 最小验收用例

未完成设计文档，不进入编码。

### 2.2 服务边界优先

不得因为“文件少”就把多个服务混在一起。

也不得为了展示微服务而拆出没有独立业务意义的服务。

判断一个服务是否应该独立，需要看：

- 是否拥有独立业务实体
- 是否拥有独立状态流转
- 是否有独立扩展需求
- 是否有独立故障边界
- 是否有独立权限或审计要求

### 2.3 平台核心由 C++ 服务维护

平台核心数据包括：

- 用户
- 团队
- 项目
- 经验记录
- Skill 草稿
- 审核记录
- 已发布 Skill
- Skill 版本
- 文件产物元信息

这些数据必须由 C++ 服务维护，Python Worker 不直接修改核心业务表，除非通过受控回写接口。

### 2.4 Agent/RAG 只做辅助决策

Agent/RAG 负责：

- 生成草稿
- 提取摘要
- 检查风险
- 提示重复
- 给审核人提供参考

Agent/RAG 不负责：

- 自动批准
- 自动发布
- 绕过审核
- 直接修改已发布版本

## 三、代码目录规范

后续实现必须使用如下目录结构：

```text
services/
  gateway-service/
  identity-service/
  project-service/
  experience-service/
  skill-service/
  review-service/
  artifact-service/
workers/
  analysis-worker/
  package-worker/
  index-worker/
  notification-worker/
idl/
  proto/
  openapi/
deploy/
  docker/
  config/
docs/
  服务端文档/
  子服务文档/
scripts/
cmake/
tests/
```

每个 C++ 子服务内部建议结构：

```text
service-name/
  CMakeLists.txt
  include/
  source/
    main.cc
    server/
    handler/
    service/
    dao/
    model/
    config/
  config/
  docs/
  tests/
```

## 四、命名规范

### 4.1 服务命名

服务目录统一使用：

```text
xxx-service
```

例如：

```text
skill-service
review-service
artifact-service
```

### 4.2 接口命名

HTTP API 使用 REST 风格：

```text
/api/v1/skill-drafts/{draft_id}/submit
/api/v1/skills/{skill_id}/versions/{version}
```

### 4.3 数据库命名

- 表名：小写下划线，复数
- 字段名：小写下划线
- 状态字段：统一使用 `status`
- 创建时间：`created_at`
- 更新时间：`updated_at`

### 4.4 消息命名

Routing Key 使用：

```text
领域.动作
```

例如：

```text
analysis.generate_skill
package.build_skill
index.skill
notification.review
```

## 五、接口设计约束

### 5.1 统一响应格式

所有接口必须返回统一结构：

```json
{
  "code": "OK",
  "message": "success",
  "request_id": "req_xxx",
  "data": {}
}
```

### 5.2 请求 ID

每个请求必须生成 `request_id`，并贯穿：

- gateway 日志
- RPC 调用
- MQ 消息
- Worker 日志
- 错误响应

### 5.3 错误码

错误码必须归类：

```text
AUTH_
VALIDATION_
PROJECT_
EXPERIENCE_
SKILL_
REVIEW_
ARTIFACT_
ANALYSIS_
SYSTEM_
```

不得直接向用户返回底层异常文本。

## 六、数据库约束

### 6.1 发布版本不可变

`skill_versions` 中已经发布的版本不可修改。

如需变更，必须创建新版本。

### 6.2 审核记录不可删除

审核记录属于审计数据，不做物理删除。

### 6.3 文件必须校验

上传文件必须记录：

- 文件名
- 文件大小
- content type
- sha256
- FastDFS key

### 6.4 状态流转必须受控

例如 Skill 草稿状态只能按以下方式流转：

```text
draft -> submitted
submitted -> needs_changes
needs_changes -> submitted
submitted -> approved
submitted -> rejected
approved -> published
```

不得随意跳转。

## 七、MQ 约束

### 7.1 消息必须带 trace_id

所有消息必须包含：

- event_id
- event_type
- trace_id
- occurred_at
- producer
- payload

### 7.2 消费者必须幂等

使用 `event_id` 或 `job_id` 做幂等判断。

### 7.3 失败必须可追踪

Worker 失败后必须回写：

- 任务状态
- 错误摘要
- 失败时间
- 重试次数

不得只在日志中记录。

## 八、安全约束

### 8.1 敏感信息

以下内容不得进入已发布 Skill：

- 明文密码
- token
- 私钥
- 内部账号
- 未脱敏客户信息
- 未授权内部链接

### 8.2 脚本安全

Skill 中的 scripts 必须经过审核。

高风险命令必须被标记：

```text
rm -rf
chmod 777
curl | sh
sudo
git reset --hard
docker system prune
```

不是所有高风险命令都禁止，但必须明确用途和适用范围。

## 九、验收约束

每个子服务交付前必须具备：

- 子服务设计文档
- API/RPC 定义
- 数据模型说明
- 错误码说明
- 最小运行说明
- 单元或接口测试说明
- 日志和配置说明
