# 02. 项目设计

## 一、设计目标

本项目采用 C++ 微服务架构实现平台核心能力，使用 Python/LangGraph 实现 RAG 和 Agent 分析工作流。

设计目标：

1. 服务边界清晰，职责单一。
2. 平台核心数据由 C++ 服务维护。
3. Agent 分析任务异步执行，不阻塞主流程。
4. 所有 Skill 发布行为可追溯。
5. 所有自动生成内容必须经过审核后才能发布。
6. 代码结构按企业级服务目录组织，不在根目录随意堆放。

## 二、整体架构

```text
前端 / CLI / API 调用方
        |
        v
gateway-service
        |
        +--> identity-service
        +--> project-service
        +--> experience-service
        +--> skill-service
        +--> review-service
        +--> artifact-service
        |
        +--> RabbitMQ
                |
                +--> analysis-worker
                +--> package-worker
                +--> index-worker
                +--> notification-worker
```

## 三、子服务划分

### 3.1 gateway-service

职责：

- 对外 HTTP API
- 鉴权
- 参数基础校验
- 请求转发
- 响应格式统一
- 请求 ID / trace ID 生成

### 3.2 identity-service

职责：

- 用户管理
- 团队管理
- 角色管理
- API Key 管理
- 权限判断

### 3.3 project-service

职责：

- 项目创建和管理
- 项目策略配置
- 项目与 Skill、经验、产物的归属关系

### 3.4 experience-service

职责：

- 经验记录提交
- 来源信息管理
- 附件关联
- 触发自动分析任务

### 3.5 skill-service

职责：

- Skill 草稿管理
- Skill 状态流转
- 已发布 Skill 管理
- Skill 版本管理
- 废弃和回滚

### 3.6 review-service

职责：

- 审核队列
- 审核意见
- 审核状态流转
- 审核审计日志

### 3.7 artifact-service

职责：

- 文件上传
- 文件下载
- FastDFS 存储映射
- 文件校验和
- Skill 包产物管理

### 3.8 worker 服务

职责：

- analysis-worker：RAG 检索、Skill 生成、安全检查、重复检查
- package-worker：生成 Skill 安装包
- index-worker：写入 Elasticsearch
- notification-worker：审核通知、任务结果通知

## 四、技术选型

### 4.1 C++ 后端

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| HTTP | cpp-httplib | 对外接口 |
| RPC | brpc + protobuf | 内部服务调用 |
| 注册中心 | etcd | 服务注册发现 |
| 数据库 | MySQL + ODB | 业务数据 |
| 缓存 | Redis++ | 状态缓存、限流、锁 |
| 消息队列 | RabbitMQ / AMQP-CPP | 异步任务 |
| 文件存储 | FastDFS | 附件、Skill 包 |
| 搜索 | Elasticsearch | Skill/经验检索 |
| 日志 | spdlog | 服务日志 |
| 参数配置 | gflags | 启动参数 |

### 4.2 Agent/RAG

| 类型 | 技术 | 用途 |
| --- | --- | --- |
| 工作流 | LangGraph | 多步骤分析流程 |
| 文档处理 | Python | PDF/Markdown/文本解析 |
| RAG | Elasticsearch + Embedding 后续扩展 | 相似 Skill、经验检索 |
| 模型调用 | 可配置 | 不绑定单一模型 |

## 五、通信设计

| 调用方向 | 通信方式 |
| --- | --- |
| 客户端 -> gateway | HTTP + JSON |
| gateway -> C++ 子服务 | brpc |
| C++ 子服务之间 | brpc |
| 子服务 -> 异步任务 | RabbitMQ |
| worker -> 平台回写 | HTTP 回调或 brpc 网关适配 |
| 服务注册发现 | etcd |

## 六、错误与异常设计

统一响应结构：

```json
{
  "code": "OK",
  "message": "success",
  "request_id": "req_xxx",
  "data": {}
}
```

错误码分类：

| 前缀 | 说明 |
| --- | --- |
| AUTH_ | 登录、鉴权、权限错误 |
| VALIDATION_ | 参数校验错误 |
| PROJECT_ | 项目相关错误 |
| EXPERIENCE_ | 经验记录错误 |
| SKILL_ | Skill 生命周期错误 |
| REVIEW_ | 审核流程错误 |
| ARTIFACT_ | 文件产物错误 |
| ANALYSIS_ | 自动分析错误 |
| SYSTEM_ | 系统内部错误 |

## 七、代码目录规划

后续代码必须按以下结构组织：

```text
services/
  gateway-service/
  identity-service/
  project-service/
  experience-service/
  skill-service/
  review-service/
  artifact-service/
workers/
  analysis-worker/
  package-worker/
  index-worker/
  notification-worker/
idl/
  proto/
  openapi/
deploy/
  docker/
  config/
docs/
  服务端文档/
  子服务文档/
scripts/
cmake/
third_party/
tests/
```

禁止将业务代码直接放在仓库根目录。
