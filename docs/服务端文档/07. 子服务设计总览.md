# 07. 子服务设计总览

## 一、文档目的

本文用于定义各子服务的设计边界和后续详细文档拆分方式。

每个子服务后续都应在 `docs/子服务文档/` 下拥有独立文档，类似视频点播系统中的“用户子服务、文件子服务、网关子服务”等。

## 二、子服务列表

| 编号 | 子服务 | 说明 | 优先级 |
| --- | --- | --- | --- |
| 01 | gateway-service | 对外 HTTP 网关 | P0 |
| 02 | identity-service | 用户、团队、角色、权限 | P0 |
| 03 | project-service | 项目管理与项目策略 | P0 |
| 04 | experience-service | 经验提交和分析任务触发 | P0 |
| 05 | skill-service | Skill 草稿、发布、版本 | P0 |
| 06 | review-service | 审核队列、审核意见、审计 | P0 |
| 07 | artifact-service | 文件上传、下载、Skill 包产物 | P1 |
| 08 | analysis-worker | RAG/Agent 自动分析 | P1 |
| 09 | package-worker | Skill 安装包构建 | P1 |
| 10 | index-worker | ES 索引写入 | P2 |
| 11 | notification-worker | 通知任务 | P2 |

## 三、第一阶段最小闭环

第一阶段只追求跑通核心业务闭环：

```text
创建项目
创建用户
提交经验
生成 Skill 草稿
提交审核
审核通过
发布 Skill
查询 Skill 仓库
```

对应服务：

- gateway-service
- identity-service
- project-service
- experience-service
- skill-service
- review-service

artifact-service 和 worker 可以先设计接口，后续逐步实现。

## 四、服务依赖关系

```text
gateway-service
  -> identity-service
  -> project-service
  -> experience-service
  -> skill-service
  -> review-service
  -> artifact-service

experience-service
  -> RabbitMQ: analysis.generate_skill

skill-service
  -> RabbitMQ: package.build_skill
  -> RabbitMQ: index.skill

review-service
  -> RabbitMQ: notification.review
```

## 七、实现顺序建议

### 7.1 第一批

1. gateway-service 基础框架
2. identity-service 用户和团队
3. project-service 项目管理

原因：

- 先确定统一入口
- 先确定用户和项目归属
- 后续所有业务都依赖项目和用户

### 7.2 第二批

1. experience-service
2. skill-service
3. review-service

原因：

- 形成核心业务主线
- 不依赖复杂 Agent 能力也能先跑通状态流转

### 7.3 第三批

1. artifact-service
2. analysis-worker
3. package-worker

原因：

- 增强文件能力
- 接入 RAG/Agent
- 生成 Skill 包

### 7.4 第四批

1. index-worker
2. notification-worker
3. 过期检测能力

原因：

- 搜索和通知属于体验增强
- 可以在核心闭环稳定后补充
