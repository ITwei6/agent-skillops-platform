# 10. 经验转Skill规则设计

## 一、文档目的

不是所有经验都值得转成 Skill。

如果平台不加筛选，团队 Skill 仓库会变成低质量内容堆积，反而降低效率。

本文定义经验转 Skill 的判断标准、生成流程和拒绝规则。

## 二、经验来源

平台支持以下经验来源：

| 来源 | 说明 |
| --- | --- |
| codex_session | Codex 工作总结 |
| cursor_session | Cursor 工作总结 |
| claude_code_session | Claude Code 工作总结 |
| langgraph_run | LangGraph Agent 执行结果 |
| pr_summary | PR 总结 |
| issue_solution | Issue 解决过程 |
| manual_note | 人工经验 |
| incident_review | 故障复盘 |

## 三、可转化标准

满足以下任一条件，才建议转成 Skill：

1. 问题或流程具有重复出现的可能。
2. 经验可以被其他团队成员复用。
3. 经验有明确输入、步骤和输出。
4. 经验解决了真实问题，并有验证结果。
5. 经验沉淀了项目规范或团队流程。
6. 经验可以减少新人学习成本。
7. 经验可以减少高风险操作失误。

## 四、不建议转化的内容

以下内容不建议转成 Skill：

### 4.1 一次性上下文

例如：

```text
今天这个临时分支怎么改。
```

### 4.2 未验证猜测

例如：

```text
可能是 Redis 的问题，感觉重启一下就行。
```

### 4.3 个人偏好

例如：

```text
我喜欢这样命名变量。
```

### 4.4 过于宽泛的建议

例如：

```text
写代码时要注意质量。
```

### 4.5 含敏感信息的原文

例如：

```text
服务器密码是 xxx。
```

### 4.6 已有 Skill 完全覆盖

如果已有 Skill 已覆盖该场景，应建议关联、反馈或更新已有 Skill，而不是新建。

## 五、转化流程

```text
经验提交
  -> 内容清洗
  -> 敏感信息初筛
  -> 判断是否有复用价值
  -> 推荐 Skill 类型
  -> 检索相似 Skill
  -> 生成 Skill 草稿
  -> 生成自动检查报告
  -> 提交人确认
  -> 技术审核
```

## 六、内容清洗规则

生成 Skill 草稿前，应先进行内容清洗：

- 删除一次性寒暄
- 删除无关对话
- 脱敏账号密码
- 替换 token
- 抽取稳定步骤
- 保留可验证依据
- 标记不确定结论

## 七、Skill 草稿生成规则

生成草稿时，必须产出：

```text
Skill 名称
Skill 类型
description
When To Use
Inputs
Workflow
Output
Quality Checks
Boundaries
References
Risk Notes
```

## 八、相似 Skill 处理

自动分析应检索已有 Skill。

结果分类：

### 8.1 完全重复

处理：

- 不建议创建新 Skill
- 建议给已有 Skill 添加反馈或新版本

### 8.2 部分重叠

处理：

- 提示 reviewer 判断是否合并
- 可生成新 Skill，但必须说明差异

### 8.3 互补关系

处理：

- 可以创建新 Skill
- 建议在 references 中关联已有 Skill

### 8.4 无明显相似

处理：

- 正常进入审核

## 九、提交人自检

提交人提交前应回答：

```text
1. 这个经验是否会再次出现？
2. 别人能否按这个 Skill 复用？
3. 是否已经验证过？
4. 是否包含敏感信息？
5. 是否和已有 Skill 重复？
6. 是否说明了适用范围？
7. 是否说明了不适用范围？
```

## 十、转化质量指标

平台可统计：

- 经验转 Skill 比例
- 自动生成草稿通过率
- 审核驳回率
- 重复 Skill 命中率
- 发布后使用次数
- 使用反馈评分
- 被标记过期次数

这些指标用于评估平台和 Agent 生成质量。
