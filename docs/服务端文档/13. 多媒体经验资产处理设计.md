# 13. 多媒体经验资产处理设计

## 一、设计背景

团队经验不一定只来自文本。

在真实研发场景中，很多有价值的经验来自：

- 排错录屏
- 操作演示视频
- 会议录音
- 技术分享录像
- 屏幕录制 + 语音讲解
- 部署流程演示

这些内容如果只作为附件保存，复用价值较低。平台可以通过多媒体处理能力，把视频和音频转化为可检索、可分析、可生成 Skill 的经验资产。

因此，本项目规划引入 FFmpeg 能力，但定位不是视频点播，而是：

```text
多媒体经验材料处理
```

## 二、和视频点播的区别

视频点播系统关注：

- 视频上传
- 视频转码
- HLS 切片
- 播放地址
- 播放量
- 视频审核

本项目关注：

- 上传经验录屏或音频
- 解析媒体元信息
- 生成封面
- 提取音频
- 标准化媒体格式
- 为 ASR 和 Agent 分析提供输入
- 生成 Skill 草稿

本项目不做公开视频播放平台。

## 三、业务价值

多媒体经验处理可以支持以下场景：

### 3.1 排错录屏转 Skill

开发者录制一次环境排错过程。

平台处理：

```text
上传录屏
  -> 提取音频
  -> 语音转文字
  -> 提取关键步骤
  -> 生成 troubleshooting 类型 Skill 草稿
```

### 3.2 技术分享沉淀

团队分享会录音或录像上传后：

```text
提取文字
  -> 总结核心知识
  -> 生成 onboarding/documentation 类型 Skill 草稿
```

### 3.3 操作流程标准化

部署或开发流程录屏上传后：

```text
识别操作步骤
  -> 抽取前置条件和验证方式
  -> 生成 workflow/deployment 类型 Skill 草稿
```

## 四、功能范围

### 4.1 P0 不做

第一阶段不实现多媒体处理，避免影响核心 Skill 审核闭环。

P0 专注：

```text
文本经验 -> Skill 草稿 -> 审核 -> 发布
```

### 4.2 P1 支持媒体元信息

P1 支持：

- 上传视频/音频附件
- 获取媒体时长
- 获取编码信息
- 获取分辨率
- 获取文件大小
- 判断媒体类型
- 保存媒体元数据

### 4.3 P2 支持媒体预处理

P2 支持：

- 视频生成封面
- 提取音频
- 转成统一音频格式
- 转成统一视频预览格式
- 生成低码率预览文件

### 4.4 P3 支持 ASR + Agent 分析

P3 支持：

- 音频转文字
- 文字切片
- RAG 检索
- Agent 总结
- 自动生成 Skill 草稿

## 五、技术选型

### 5.1 FFmpeg 用途

使用 FFmpeg 处理：

- 媒体探测
- 音频提取
- 封面生成
- 格式转换
- 预览文件生成

### 5.2 脚手架关系

原始 C++ 脚手架包含 FFmpeg/HLS 能力。

本项目不直接复用视频点播的 HLS 业务逻辑，而是改造为更通用的媒体处理能力：

```text
media_probe
extract_audio
generate_thumbnail
transcode_preview
```

后续自研脚手架中建议加入可选模块：

```cmake
option(SKILLOPS_ENABLE_MEDIA "Enable FFmpeg media utilities" OFF)
```

默认 P0 可以关闭，等进入 P1/P2 再开启。

## 六、服务设计

### 6.1 一期归属

多媒体文件仍归 `artifact-service` 管理。

artifact-service 负责：

- 上传文件
- 保存原始文件
- 记录文件类型
- 创建媒体处理任务
- 关联处理产物

### 6.2 Worker 设计

新增规划 worker：

```text
media-worker
```

职责：

- 消费媒体处理任务
- 调用 FFmpeg
- 写入媒体元信息
- 生成封面/音频/预览文件
- 回写处理状态

### 6.3 后续拆分 media-service

如果媒体处理能力复杂化，可以从 artifact-service 中拆出：

```text
media-service
```

拆分条件：

- 媒体任务量大
- 需要独立扩容
- 需要独立管理媒体状态
- 需要提供复杂媒体查询接口

P0/P1 不建议提前拆分。

## 七、业务流程

### 7.1 上传媒体附件

```text
用户上传媒体文件
  -> artifact-service 保存文件
  -> 判断 content_type
  -> 如果是媒体文件，创建 media.process 消息
  -> 返回 artifact_id
```

### 7.2 处理媒体文件

```text
media-worker 消费 media.process
  -> 下载或读取原文件
  -> FFmpeg probe
  -> 保存时长、编码、分辨率
  -> 生成封面
  -> 提取音频
  -> 保存处理产物
  -> 更新处理状态
```

### 7.3 生成 Skill 草稿

```text
音频提取完成
  -> ASR 任务
  -> 得到转写文本
  -> analysis-worker 生成经验摘要
  -> 生成 Skill 草稿
```

## 十、文件产物设计

一个原始媒体文件可能产生多个产物：

```text
原始视频 artifact
  -> 缩略图 artifact
  -> 提取音频 artifact
  -> 预览视频 artifact
  -> ASR 文本 artifact
```

这些都由 artifact-service 统一管理。

## 十一、安全与限制

### 11.1 文件大小限制

一期建议限制：

```text
视频最大 500MB
音频最大 100MB
```

具体数值后续可配置。

### 11.2 文件类型限制

允许：

```text
mp4
mov
mkv
mp3
wav
m4a
```

不允许执行型文件伪装成媒体文件。

### 11.3 处理超时

媒体处理任务必须设置超时。

超时后：

- 标记 failed
- 记录错误
- 不阻塞经验提交主流程

## 十二、验收标准

P1 验收：

```text
上传视频后，平台能记录时长、编码、分辨率。
```

P2 验收：

```text
上传视频后，平台能生成封面并提取音频。
```

P3 验收：

```text
上传排错录屏后，平台能基于转写文本生成 Skill 草稿。
```
