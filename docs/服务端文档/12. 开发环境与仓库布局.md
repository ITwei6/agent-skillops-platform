# 12. 开发环境与仓库布局

## 一、文档目的

本文说明开发环境中各类代码应该放在哪里，以及 C++ 微服务脚手架应该如何编译和安装。

本项目后续实现必须区分：

- 开发环境仓库
- C++ 微服务脚手架仓库
- Agent SkillOps 业务项目仓库

不要把业务代码写进脚手架仓库，也不要把脚手架源码直接复制进业务项目。

## 二、三类仓库的关系

### 2.1 dev-environment

开发环境仓库，负责启动 Docker 容器和中间件。

包含：

- dev-env-service
- MySQL
- Redis
- RabbitMQ
- Elasticsearch
- Kibana
- Etcd
- FastDFS

它不是业务项目源码。

### 2.2 cpp-microservice-scaffold

C++ 微服务脚手架仓库。

作用：

- 封装日志
- 封装 RPC
- 封装 MQ
- 封装 Redis
- 封装 MySQL/ODB
- 封装 FastDFS
- 封装 HLS/FFmpeg
- 提供 CMake 包

它应该作为基础库编译安装到开发容器中。

### 2.3 agent-skillops-platform

本项目业务仓库。

作用：

- 编写 SkillOps 平台服务端代码
- 编写接口、数据库、MQ、子服务实现
- 调用已安装好的 `bite_scaffold` 基础库

业务代码应该写在这个仓库中。

## 三、推荐目录布局

进入 Docker 开发容器后，统一在：

```bash
/home/dev/workspace
```

下工作。

推荐布局：

```text
/home/dev/workspace/
  third_party/
    cpp-microservice-scaffold/
  projects/
    agent-skillops-platform/
```

说明：

- `third_party/cpp-microservice-scaffold`：只放脚手架源码，用于编译安装。
- `projects/agent-skillops-platform`：本项目业务代码，后续主要开发目录。

不要在 `cpp-microservice-scaffold` 里写 SkillOps 业务代码。

## 四、脚手架编译位置

脚手架应该在 Docker 开发容器内部编译，而不是在 Windows 上编译。

原因：

- 脚手架依赖大量 Linux C++ 库。
- Docker 开发环境已经预装依赖。
- 后续业务项目也在容器中编译运行。

## 五、脚手架编译步骤

进入开发容器：

```bash
ssh dev@127.0.0.1 -p 22222
```

进入工作目录：

```bash
cd ~/workspace
mkdir -p third_party projects
```

克隆脚手架：

```bash
git clone https://gitee.com/bitedu-cpp-team/cpp-microservice-scaffold.git third_party/cpp-microservice-scaffold
```

编译安装：

```bash
cd ~/workspace/third_party/cpp-microservice-scaffold
mkdir -p build
cd build
cmake .. -DCMAKE_INSTALL_PREFIX=/usr
make -j$(nproc)
sudo make install
```

安装完成后，容器中应存在：

```text
/usr/include/bite_scaffold/
/usr/lib/
/usr/lib/cmake/bite_scaffold/
```

业务项目后续通过 CMake 使用：

```cmake
find_package(bite_scaffold REQUIRED)
target_link_libraries(target_name PRIVATE bite_scaffold::bite_scaffold)
```

## 六、业务项目代码位置

业务项目应克隆到：

```bash
~/workspace/projects/agent-skillops-platform
```

推荐命令：

```bash
cd ~/workspace/projects
git clone git@github.com:ITwei6/agent-skillops-platform.git
```

后续 VS Code Remote SSH 连接进入容器后，应打开：

```text
/home/dev/workspace/projects/agent-skillops-platform
```

而不是打开：

```text
/home/dev/workspace/third_party/cpp-microservice-scaffold
```

## 七、业务项目代码组织

业务项目后续代码应放在：

```text
services/
workers/
idl/
deploy/
scripts/
tests/
```

其中 C++ 服务放：

```text
services/gateway-service/
services/identity-service/
services/project-service/
services/experience-service/
services/skill-service/
services/review-service/
services/artifact-service/
```

Python Worker 放：

```text
workers/analysis-worker/
workers/package-worker/
workers/index-worker/
workers/notification-worker/
```

## 八、和视频点播项目的关系

视频点播项目的 `server` 目录是业务服务端代码目录。

本项目也应该类似：

```text
agent-skillops-platform/
  services/
  workers/
  idl/
  docs/
  deploy/
```

区别是：

- 视频点播的业务是用户、文件、视频、转码、网关。
- 本项目的业务是经验、Skill、审核、发布、产物、分析。

脚手架只是基础库，不是业务项目本体。

## 九、开发顺序建议

### 9.1 第一步：准备开发环境

```bash
cd ~/Desktop/dev-environment
docker compose up -d
```

### 9.2 第二步：进入开发容器

```bash
ssh dev@127.0.0.1 -p 22222
```

### 9.3 第三步：编译安装脚手架

```bash
cd ~/workspace
mkdir -p third_party projects
git clone https://gitee.com/bitedu-cpp-team/cpp-microservice-scaffold.git third_party/cpp-microservice-scaffold
cd third_party/cpp-microservice-scaffold
mkdir -p build && cd build
cmake .. -DCMAKE_INSTALL_PREFIX=/usr
make -j$(nproc)
sudo make install
```

### 9.4 第四步：克隆业务项目

```bash
cd ~/workspace/projects
git clone git@github.com:ITwei6/agent-skillops-platform.git
```

### 9.5 第五步：用 VS Code 打开业务项目

VS Code Remote SSH 连接后打开：

```text
/home/dev/workspace/projects/agent-skillops-platform
```

## 十、关键原则

1. 脚手架只编译安装，不写业务。
2. 业务代码只写在 `agent-skillops-platform`。
3. 所有 C++ 服务都通过 `find_package(bite_scaffold REQUIRED)` 引入脚手架。
4. Docker 开发容器是主要编译运行环境。
5. Windows 只作为编辑器宿主，不作为 C++ 服务编译环境。
