# 04. API协作与状态管理

## 一、文档目的

本文定义前端与后端 API 的协作方式、请求封装、状态管理和错误处理。

## 二、API 基础约定

后端统一通过 `gateway-service` 提供 HTTP API。

前端只调用：

```text
VITE_API_BASE_URL
```

不得在前端直接调用内部 C++ 子服务。

## 三、统一响应结构

后端响应：

```json
{
  "code": "OK",
  "message": "success",
  "request_id": "req_001",
  "data": {}
}
```

前端处理规则：

- `code === "OK"`：返回 `data`
- `code !== "OK"`：抛出业务错误并提示 `message`
- 记录 `request_id`，方便排查

## 四、Axios 封装

建议文件：

```text
src/api/request.ts
```

封装能力：

- baseURL
- token 注入
- request_id 透传，可后续扩展
- 统一错误处理
- loading 可选
- 文件上传支持

## 五、API 模块划分

```text
src/api/
  identity.ts
  project.ts
  experience.ts
  skillDraft.ts
  skill.ts
  review.ts
  artifact.ts
  analysis.ts
```

不要在页面里直接写 axios URL。

## 六、类型定义

建议文件：

```text
src/types/domain.ts
src/types/api.ts
```

核心类型：

```ts
type UserRole = 'developer' | 'reviewer' | 'maintainer' | 'admin'
type SkillDraftStatus = 'draft' | 'submitted' | 'needs_changes' | 'approved' | 'rejected' | 'published'
type RiskLevel = 'low' | 'medium' | 'high' | 'blocked'
type AnalysisJobStatus = 'queued' | 'running' | 'succeeded' | 'failed'
```

## 七、状态管理

使用 Pinia。

### 7.1 user store

保存：

- token
- 当前用户
- 角色
- 权限

### 7.2 app store

保存：

- 当前项目
- 侧边栏状态
- 全局 loading

### 7.3 dict store

保存：

- 状态枚举
- 风险等级
- Skill 类型
- 来源类型

## 八、权限控制

前端需要做路由级控制，但最终权限以后端为准。

前端规则：

| 页面 | 可访问角色 |
| --- | --- |
| 工作台 | 全部登录用户 |
| 经验提交 | developer/reviewer/maintainer/admin |
| 审核中心 | reviewer/maintainer/admin |
| Skill 发布 | maintainer/admin |
| 系统管理 | admin |

如果用户无权限：

- 菜单隐藏
- 直接访问路由时显示无权限页

## 九、数据刷新策略

### 9.1 列表页

操作成功后刷新当前列表。

### 9.2 详情页

状态变更后重新拉取详情。

### 9.3 分析任务

分析任务可轮询：

```text
每 3 秒查询一次
成功或失败后停止
超过 5 分钟停止并提示
```

后续可改成 WebSocket 或 SSE，但一期不做。

## 十、错误处理

### 10.1 业务错误

显示后端 message。

### 10.2 登录失效

清理 token，跳转登录页。

### 10.3 网络错误

提示：

```text
网络异常，请稍后重试。
```

### 10.4 系统错误

提示：

```text
系统异常，请联系管理员，并提供 request_id。
```
