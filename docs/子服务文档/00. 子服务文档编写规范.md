# 00. 子服务文档编写规范

## 一、目的

子服务文档用于指导后续实现窗口进行编码。

每个子服务在编码前必须先有独立设计文档，避免代码阶段临时决定服务职责、接口、表结构和消息格式。

## 二、文档结构

每份子服务文档统一包含以下章节：

```text
一、服务职责
二、模块划分
三、接口设计
四、数据设计
五、缓存设计
六、消息队列设计
七、核心业务流程
八、错误码设计
九、配置项设计
十、验收标准
```

## 三、服务职责写法

必须说明：

- 这个服务负责什么
- 不负责什么
- 依赖哪些服务
- 被哪些服务调用

不能只写一句“负责用户管理”。

## 四、模块划分写法

建议按企业后端分层：

```text
handler 层：处理请求和响应
service 层：业务逻辑
dao 层：数据库访问
model 层：实体模型
client 层：调用其他服务
config 层：配置解析
```

## 五、接口设计写法

接口必须包含：

- URL 或 RPC 名称
- 请求参数
- 响应参数
- 错误码
- 权限要求
- 状态变更说明

## 六、数据设计写法

必须说明：

- 使用哪些表
- 读写哪些字段
- 是否需要事务
- 是否需要唯一索引
- 是否涉及审计数据

## 七、缓存设计写法

如果不使用缓存，也必须写：

```text
本服务一期不使用 Redis 缓存。
```

不能空着。

## 八、MQ 设计写法

如果服务生产消息，必须写：

- Exchange
- Routing Key
- Payload
- 触发时机
- 幂等要求

如果服务消费消息，必须写：

- Queue
- 消费者逻辑
- 重试策略
- 失败处理

## 九、验收标准写法

验收标准必须可验证。

例如：

```text
给定一个 draft 状态的 Skill 草稿，调用提交审核接口后，状态必须变为 submitted，并生成一条审核待办。
```

不要写：

```text
功能正常即可。
```
